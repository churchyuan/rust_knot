# 修改 3_1 纽结臂型判定算法计划 (更新)

## 摘要 (Summary)
之前的方法（使用射线法检测交叉点 2 是否在另外两条路径组成的多边形内部）在测试中报错：“Expected exactly 2 outer paths, found 4”。
**原因分析：**
用户指出，由于原始链可能是一个环（Ring），纽结核心可能会跨越原始数组的边界（例如 90 -> 100 -> 1 -> 20）。在现有的 `passes_through` 和多边形构建逻辑中，我们简单地使用了 `min_p..max_p` 进行区间判定，完全忽略了环形拓扑的**取模回绕（wrap-around）**，导致路径截取和点包含判定全部失效。
此外，判断臂型需要**依赖纽结在整条链上的位置**，因此必须回到整条链（Full Chain）上，基于模运算（Modulo arithmetic）处理所有的区间和距离。

**修正方案：**
1. 修改 `get_31knot_arm_type` 的函数签名：除了传入 `core_points`，还需要传入原始的**整条链的长度 `full_chain_len`**（因为核心点里的 index 依然是基于整条链的）。
2. 在环状拓扑下，任意两个在链上的点 $A$ 和 $B$ 之间实际上存在**两条**可能的路径：
   - 顺时针（$A \to B$）：长度为 $(B - A + N) \pmod N$
   - 逆时针（$A \to B$ 的反向，即 $B \to A$）：长度为 $(A - B + N) \pmod N$
3. 因此，交叉点 1 (locs1: [A, B]) 和 交叉点 3 (locs3: [C, D]) 之间可以形成 4 种点对组合。对于每一个点对组合（例如 A 和 C），在环上都有**顺时针和逆时针两条路径**。这样总共有 8 条潜在路径。
4. **路径生成与提取：** 
   - 既然是环，我们只考虑**在纽结核心内部**的路径。因为 1, 2, 3 都是在核心上的交叉点，连接 1 和 3 的路径不应该绕过整条长链的非纽结部分（这部分通常很长，不属于纽结形态）。
   - **关键修正**：与其在全链上找 8 条路径，不如利用核心本身是连续提取出来的（比如从 `core.left` 到 `core.right`）。我们在核心的这段局部序列上，可以安全地将原始环索引**展开（unwrap）**成单调递增的虚拟索引（例如 90, 91, ..., 99, 100, 101(原1), 102(原2)...），这样就完美避开了模运算带来的区间断裂问题。

## 拟议更改 (Proposed Changes)

### 1. `src/arm_type.rs`
**修改 `get_31knot_arm_type` 逻辑：**
- **虚拟索引映射 (Unwrapping Indices)：** 
  不再直接使用 `enumerate()` 赋予的绝对递增索引。在 `get_31knot_arm_type` 内部，我们将核心段 `core_points` 的原始索引（通过 `points.iter().enumerate()` 携带）进行“解卷（Unwrap）”：
  - 设核心第一个点的虚拟索引为 $v_0 = 0$。
  - 遍历核心序列，根据原始索引 $i_k$ 与 $i_{k-1}$ 的差值：如果 $i_k$ 相比 $i_{k-1}$ 发生突降（例如从 99 变到 0），说明跨越了环边界，虚拟索引应继续累加（例如 $v_k = v_{k-1} + 1$）。由于我们传入的 `core_points` 是连续提取的，因此虚拟索引 $v$ 永远是 $0, 1, 2, ..., \text{len}-1$。
  - **等等，我们甚至不需要原始索引！** 因为 `core_points` 就是按顺序连续排列的核心段！我们直接使用 `core_points` 在数组中的下标（$0$ 到 $\text{len}-1$）作为区间判定的依据即可！
  - 之前出错的原因恰恰是因为我们在 `arm_type.rs` 中使用 `enumerate()` 时，**错误地将提取出来的核心数组的下标 (0, 1, 2...) 当作了原始链的 Index，并且在 KMT 简化后依然使用这个 0..N-1 的相对下标**。这在逻辑上是完全成立的，且天然不受 Ring 边界影响（因为它已经被展平为数组）。
  - **那么为什么会报错 `Expected exactly 2 outer paths, found 4` 呢？**
    - 回顾报错：所有的 4 条路径都 `!passes_through(path, locs2)`。
    - 这意味着交叉点 2 的下标（例如 [E, F]）不在 [A, C], [A, D], [B, C], [B, D] 的**任何一个开区间** `min_p < x < max_p` 内。
    - **这说明排序（步骤 8）出了问题！** 
      我们目前的排序是将所有 6 个位置（交叉点 1、2、3 的各 2 个位置）放在一起排序，然后按首次出现将交叉点命名为 1、2、3。
      这会导致交叉点 2 永远夹在 1 和 3 之间，例如 1-2-3-3-2-1。
      如果拓扑是 1-2-3-1-3-2：
      locs1: [10, 40]
      locs2: [20, 60]
      locs3: [30, 50]
      按首次出现排序：1(10), 2(20), 3(30)。所以交叉点命名依然是 1, 2, 3。
      但是！对于组合 `(locs1[1], locs3[1])` 即 `(40, 50)`，它的区间是 `40..50`。而 locs2 是 `[20, 60]`。
      20 不在 40..50 内，60 也不在 40..50 内！所以它不 pass_through。
      同理，对于组合 `(10, 50)`，20 在里面，所以 pass_through = true。
      结果就是，可能有 3 条甚至 4 条路径都不包含 locs2！

**重构路径判定逻辑 (基于射线法)：**
- 我们不应该强行猜测“一定有两条包含交叉点 2 的路径”。
- 我们直接在 2D 平面上，**穷举 1 和 3 之间的 4 条物理连接路径（实际上在链上就是提取这两个 index 之间的所有点）**。
- 这 4 条路径两两组合，可以构成 6 个闭合环（Polygon）。
- 我们需要找到**真正代表纽结外部轮廓（Outer Boundary）的那个环**。
- **更稳健的做法**：按照纽结的实际定义：
  - 纽结核心上有 3 个交叉点，将核心分成了 6 段圆弧（Arcs）。
  - 其中 2 个圆弧是“游离臂”（Arm），它们在 2D 投影上构成了外围的轮廓。
  - 直接在 2D 投影中找到那个最大的外围多边形，然后看交叉点 2 是否在其中。
  - 或者，回归用户的初衷：**“1-3应该有4种连接方式，选取最短的2种连接方式”**。
  - 之前代码中：
    ```rust
    let mut paths_with_dist: Vec<((usize, usize), usize)> = paths
        .into_iter()
        .map(|(a, b)| ((a, b), a.abs_diff(b)))
        .collect();
    ```
    这里 `a.abs_diff(b)` 是基于简化后链的下标。但这可能不够准确，或者正是因为这个过滤条件，导致我们之前在代码重构时把它删掉了。

### 修正策略：严格遵守用户最新逻辑
1. 找出交叉点 1 和 3 的 4 种连接方式：`[locs1[0], locs3[0]]`, `[locs1[0], locs3[1]]`, `[locs1[1], locs3[0]]`, `[locs1[1], locs3[1]]`。
2. 计算这 4 个区间的物理长度（在未简化链上的点数）。由于传入的是 `core_points`（展开后的开链），物理长度就是 `a.abs_diff(b)`。
3. **选出最短的 2 条连接方式（路径）**。
4. 将这两条最短路径上的点提取出来，在 XY 投影面上首尾相接拼成一个多边形（Polygon）。
5. 将交叉点 2 的两个位置投影到 XY 面上。
6. 使用射线法检查这两个点是否在多边形内部。
7. 如果都在内部 -> "two-arm"，否则 -> "three-arm"。

这样既保留了用户关于“最短2条路径”的初衷，又加入了最新的“多边形内部射线法判定”，同时避免了 `passes_through` 产生的 4 条 outer_paths 错误。
