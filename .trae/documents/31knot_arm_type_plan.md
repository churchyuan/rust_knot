# 修改 3_1 纽结臂型判定算法计划 (终极版)

## 摘要 (Summary)
用户确认了 3_1 纽结的臂型（two-arm/three-arm）必须在**整条环形链（Ring）**上计算，而不仅仅是在纽结核心片段上。
算法逻辑如下：
1. **交叉点映射回整链：** 纽结核心片段经过 KMT 简化后得到了交叉点 1、2、3。这 3 个交叉点在原始的完整环形链上有对应的实际 Index。
2. **构建 4 条路径：** 交叉点 1（对应 2 个点）和 交叉点 3（对应 2 个点）在**整条环上**一共有 4 条顺时针/逆时针对应的短片段（因为用户明确指出，不需要考虑长绕远路的那一段，只考虑“顺时针选取和逆时针选取是一样的”这种最直接连接 1 和 3 的线段）。
3. **筛选多边形边界：** 这 4 条路径中，必然有 2 条经过了交叉点 2 对应的点（包含交叉点 2）。我们把这 2 条**排除掉**。
4. **构建多边形：** 剩下**不包含交叉点 2 的 2 条路径**，提取它们在原始整链上的所有 3D 点，首尾相连组成一个闭合的多边形（投影到 2D 平面）。
5. **位置判定：** 最后，用射线法判定交叉点 2 对应的两个点，如果都在该多边形内部，则为 `two-arm`，否则为 `three-arm`。

## 当前状态分析 (Current State Analysis)
- 目前 `get_31knot_arm_type` 接收的是 `core_points`，导致内部无法获取整条链的信息，也无法跨越环的边界（如 `[90..10]`）。
- `passes_through` 和构建路径逻辑都只适用于线性数组，没有处理环形数组的模运算（Modulo）。
- 之前使用的“选取最短距离的两条线”存在误判，必须换回**“筛选出未经过交叉点 2 的两条线”**的逻辑。

## 拟议更改 (Proposed Changes)

### 1. `src/arm_type.rs` 的签名修改
- 将函数签名修改为接收 `full_points: &[Point3]`（完整的环链）以及 `core_start: i32`, `core_end: i32`。
- 函数内部，首先从 `full_points` 提取出连续的 `core_points` 序列，但同时**记录每一个提取出的点在 `full_points` 中的原始 Index**。
- 这意味着 `working` 初始化时，不仅包含 `Point3`，还要绑定它在 `full_points` 里的真实绝对 Index，如：
  ```rust
  let mut curr = core_start as usize;
  // loop ...
  working.push((full_points[curr], curr));
  ```
- 然后对这个 `working` 数组进行 KMT 简化。简化后剩下的点，其附带的 `usize` 就是在 `full_points` 里的真实绝对 Index。

### 2. 重写路径生成和筛选逻辑（步骤 9 和 10）
- 获得交叉点 1、2、3 的位置 `locs1`, `locs2`, `locs3` 后（此时它们是 `full_points` 里的真实 Index），构建 4 条路径。
- **路径定义：** 
  - 给定环上两点 A 和 B，在**核心片段覆盖的范围内**（即沿着 `core_start` 到 `core_end` 的方向），找出连接 A 和 B 的序列。由于 1、2、3 都在核心段内，A 和 B 的相对位置是明确的。
  - 由于提取 `core_points` 已经展平了环形边界，我们可以直接使用它们在 `core_points` 数组里的**相对下标（0 到 core_size-1）**进行区间包含的判定（即判断 2 是否在 1 和 3 之间）。
  - **核心思路转变：** 
    - 其实，判断“是否经过交叉点 2” 在**展开后的核心数组（Relative Index）**上做最简单、最准确，因为没有任何环的断层！
    - 构建多边形提取 3D 点时，直接从**展开后的核心数组**里切片拿点，也是最完整连续的！完全不需要去 `full_points` 里绕圈圈处理模运算。
    - **用户的意图：** “计算连接时，考虑的是整条 ring”。实际上，如果我们在展开的核心数组里切片（比如下标 40 到 60），它提取的 3D 点序列，与在 `full_points` 里提取对应真实 Index 的点序列是**完全一样的**（比如真实 Index 90 -> 100 -> 0 -> 10），因为核心数组本来就是从环上抄写下来的。
    - **结论：不需要修改函数签名！继续接收 `core_points`，只修改内部判定逻辑。**

### 3. 真正修正“Expected exactly 2 outer paths, found 4”的过滤逻辑
- 为什么之前的 `passes_through` 会找不出包含 2 的路径？
  - 因为 4 种连接方式是 `locs1` 和 `locs3` 之间的组合。
  - 交叉点 2 有两个位置 `E` 和 `F`。
  - 对于路径 `[A, C]`，它包含点 2，意思是：`E` 在 `A` 和 `C` 之间，**或者** `F` 在 `A` 和 `C` 之间。
  - **边界问题：** KMT 简化后的下标可能是 `10, 11, 12`。如果 `A=10, C=12, E=11`，那么 `10 < 11 < 12` 成立。但如果简化后点很少，`A=10, C=11`，那么 `E` 可能根本不在中间。
  - **更稳健的判断（基于相对顺序）：**
    在单调递增的展平下标中，1、2、3 的首次出现是有序的，但它们的另一半（第二次穿过）位置可能交错。
    对于任意区间 `[min_p, max_p]`，点 `x` 是否在里面：
    ```rust
    let passes_through = |path: (usize, usize), l2: [usize; 2]| -> bool {
        let (min_p, max_p) = (path.0.min(path.1), path.0.max(path.1));
        // 使用闭区间 <= 因为 KMT 简化可能会把交叉点挤压到同一个点
        (l2[0] >= min_p && l2[0] <= max_p) || (l2[1] >= min_p && l2[1] <= max_p)
    };
    ```
    - 之前因为用了严格的小于号 `<`，导致边界重合时返回 `false`，从而 4 条路径都不包含 2。
    - 修改为 `<=` 之后，包含交叉点 2 的路径就能被正确识别出来。

### 4. 提取与构建多边形
- 用 `<=` 过滤后，期望得到 2 条 `outer_paths`（不包含交叉点 2 的路径）。
- 从这 2 条路径的起始到终止位置，在 `points` (`core_points`) 数组中提取 3D 点，首尾相接拼成多边形。
- 将多边形和交叉点 2 的 2 个点都用 `find_max_span` 映射到同一 2D 平面（前两个坐标）。
- 使用射线法 `is_point_in_polygon` 判断，都在内部返回 `"two-arm"`，否则返回 `"three-arm"`。
